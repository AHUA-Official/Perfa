# 第一章 绪论

## 1.1 研究背景与意义

服务器性能测试是系统建设、上线验收、容量规划和故障定位中的基础工作。随着互联网业务、云计算平台和企业内部信息系统的规模扩大，单台服务器或单个服务实例的性能表现已经不再只是硬件参数问题，而是与操作系统配置、存储介质、网络链路、测试工具参数和业务负载模型共同相关。Dean 和 Barroso 在讨论大规模在线服务尾延迟问题时指出，当服务链路由多个组件共同构成时，少量慢请求也可能影响整体服务体验[1]。这一现象说明，性能测试不能只停留在“能否运行”或“单次分数高低”的层面，而应当结合测试过程、系统负载和运行环境对结果进行解释。对于服务器运维人员和系统开发人员而言，一套可重复、可追踪、可分析的性能测试流程能够减少主观经验判断带来的误差，为资源配置和系统优化提供依据。

传统服务器性能测试通常依赖多种独立工具完成。CPU、内存、磁盘和网络分别对应不同的测试程序，测试人员需要登录目标服务器，安装依赖，编写命令参数，执行测试任务，再从日志、终端输出或监控平台中整理结果。该流程在单次实验中尚可接受，但当测试对象增加、测试场景变复杂或需要多轮对比时，工具安装、参数管理、任务追踪和报告整理会消耗大量时间。更大的问题在于，性能测试结果往往脱离执行过程保存：测试工具输出一组指标，监控系统保存另一组时间序列数据，人工报告又以独立文档形式存在。三类信息之间缺少统一索引后，后续复盘难以判断某个异常数值究竟来自硬件瓶颈、测试参数不当、系统瞬时负载，还是工具执行失败。

现代服务器系统的复杂性进一步放大了这一问题。分布式系统通常由多语言服务、多台机器和多个中间件组件组成，性能异常可能由调用链上的任意环节触发。Google 的 Dapper 系统从大规模分布式服务追踪实践出发，强调了低开销、应用透明和可扩展的观测机制对理解系统行为的重要性[2]。虽然本文研究对象不是完整的分布式追踪系统，但这一思路对服务器性能测试平台仍有启发意义：测试平台不应只关心测试命令是否结束，还应记录测试任务、工具调用、监控指标、错误信息和报告结论之间的关系。只有将执行动作与运行证据绑定，性能测试结果才具备复查和解释价值。

可观测性技术的发展为性能测试平台提供了工程基础。Prometheus 将监控数据组织为时间序列，并通过指标抓取、查询和告警机制支持系统运行状态分析[3]。OpenTelemetry 从 traces、metrics 和 logs 等信号出发，为不同系统之间的可观测数据采集与导出提供统一框架[4]。这些技术说明，服务器性能测试不应仅产生离线分数，还应与持续采集的 CPU、内存、磁盘、网络等指标结合，使测试过程能够被观察、被定位和被复现。对于教学和毕业设计场景而言，将压测工具、监控指标和报告生成整合到同一平台中，也能更完整地体现软件系统设计、后端接口、前端交互和工程部署能力。

与此同时，大语言模型的发展为复杂系统操作方式带来了新的变化。Brown 等提出的大规模语言模型在少样本任务中表现出较强的泛化能力[5]，Wei 等关于思维链提示的研究表明，复杂问题可以通过中间推理步骤被拆解和求解[6]。这些研究并不意味着大语言模型可以直接替代服务器测试工具，但它们说明自然语言可以成为复杂任务规划和人机交互的入口。性能测试任务本身具有明显的过程性：用户先描述测试目标，系统需要识别测试类型、选择工具、检查环境、执行任务、获取结果并生成说明。若将大语言模型用于测试流程编排，就有可能把原本分散的命令行操作转化为对话式任务执行，从而降低使用门槛。

大语言模型用于工程系统时必须面对一个边界：模型可以生成计划和解释文本，但不能凭空获取目标服务器状态，也不能替代真实工具执行。ReAct 方法将推理过程与外部动作结合，使模型能够在思考与环境交互之间形成闭环[7]。Toolformer 研究则进一步说明，语言模型可以通过调用外部工具弥补自身在实时信息、计算和外部系统操作方面的限制[8]。因此，面向服务器性能测试的智能平台不能只设计聊天界面，而应建立一条从自然语言请求到工具调用、从工具结果到报告分析的可执行链路。模型负责理解意图和组织流程，工具层负责真实执行和结果返回，二者之间需要明确的协议边界。

Model Context Protocol（MCP）为这一边界提供了新的工程实现方式。Anthropic 发布 MCP 时将其定位为连接 AI assistant 与外部数据源、业务工具和开发环境的开放标准[9]。MCP 架构文档进一步将客户端、服务器和工具能力组织为统一协议，使 AI 应用能够以标准方式获取上下文并调用外部能力[10]。对于服务器性能测试平台而言，MCP 的价值不在于替代测试工具，而在于把服务器管理、Agent 部署、工具安装、Benchmark 执行和报告生成封装为可被上层智能体调用的工具集合。这样一来，平台可以避免在大语言模型编排层直接耦合大量系统命令，也便于后续扩展新的测试工具和运维能力。

基于上述背景，本文围绕 Perfa 项目开展研究与实现。Perfa 是一套面向服务器性能测试场景的多组件系统，目标是将“自然语言驱动的服务器性能测试”落地为可执行平台。系统由 WebUI V2、LangChain Agent、MCP Server 和 Node Agent 四个主要部分组成。WebUI V2 提供对话、服务器、报告和监控页面；LangChain Agent 负责理解用户请求、组织工作流并对外提供 OpenAI 兼容接口；MCP Server 将服务器管理、Agent 生命周期、工具管理、压测任务和报告生成封装为标准 Tool；Node Agent 部署在被测节点上，负责资源监控、工具管理、Benchmark 执行和 HTTP API 暴露。通过这种分层设计，用户可以通过自然语言发起性能测试请求，系统则将请求逐级转换为可验证的工具调用和节点执行动作。

本文的研究意义主要体现在工程实践和系统设计两个方面。在工程实践层面，Perfa 将性能测试工具、节点监控、任务编排、报告生成和可视化页面整合到统一平台中，减少了测试人员在不同工具和界面之间切换的成本。在系统设计层面，本文探索了大语言模型编排层、MCP 能力封装层和节点执行层之间的职责划分，避免将智能体直接设计为“全能执行器”。这种设计使自然语言交互与真实系统操作之间保持可审计边界，有助于提升系统的可维护性和扩展性。对于本科毕业设计而言，该课题覆盖后端服务、工具协议、节点 Agent、前端交互、监控链路和测试验证等内容，能够较完整地体现软件工程综合实践能力。

## 1.2 国内外研究现状

服务器性能测试相关研究和工具长期围绕硬件资源、操作系统行为和应用负载展开。CPU、内存、磁盘和网络分别具有不同的性能瓶颈和评价指标，因此工程实践中形成了多种面向不同资源类型的测试工具。磁盘测试通常关注 IOPS、吞吐量和延迟，内存测试关注带宽和访问延迟，网络测试关注吞吐、时延和丢包，综合性能测试则尝试从多类负载中给出整体评分。这类工具的共同特点是结果相对客观、执行成本较低，但它们通常以命令行程序或独立脚本形式存在。使用者需要理解参数含义、测试条件和输出字段，才能将工具结果转化为可用于决策的结论。

国外在性能测试和系统观测方面形成了较成熟的工程体系。Prometheus 作为开源监控系统，被广泛用于采集和查询时间序列指标[3]；OpenTelemetry 则试图在指标、日志和链路追踪之间建立统一的数据采集模型[4]。这些技术推动了性能分析从单次测试向持续观测扩展。性能测试平台如果能够在执行 Benchmark 的同时记录系统指标，就可以把“测试结果是什么”和“测试过程中系统处于什么状态”联系起来。Dapper 对大规模系统追踪的讨论也表明，复杂系统中的性能问题往往需要跨越单个程序边界进行理解[2]。这类研究为本文设计监控链路和报告证据结构提供了基础思路。

自动化运维和智能运维的发展使性能测试平台开始从工具集合走向流程系统。传统脚本可以完成固定顺序的部署和测试任务，但当用户需求发生变化时，脚本往往需要人工调整。大语言模型出现后，自然语言任务理解和流程规划成为新的技术路线。GPT-3 展示了大规模语言模型在多任务语言理解上的泛化能力[5]，思维链提示说明复杂问题可以通过中间步骤表达推理过程[6]。这些能力适合用于性能测试任务的意图识别和流程拆解，例如将“帮我测试这台服务器磁盘性能”转换为服务器选择、工具检查、fio 参数设置、任务执行和结果汇总等步骤。

围绕大语言模型与外部工具协作的研究进一步推动了智能体系统的发展。ReAct 通过将推理和动作结合，使模型能够在执行过程中根据观察结果调整后续行为[7]。Toolformer 则从工具使用角度讨论模型如何借助 API 完成其自身难以完成的任务[8]。这些研究对 Perfa 的启发在于，性能测试系统不能只依赖模型生成文本回答，而要让模型通过受控工具调用访问真实系统。测试工具的执行、服务器状态的查询和报告数据的获取都应由后端服务完成，模型只负责在合适的阶段组织调用和解释结果。这样的设计可以降低模型幻觉对系统操作的影响，也便于记录工具调用证据。

MCP 是近期 AI 工具集成方向的重要协议实践。MCP 将外部能力组织为可被 AI 应用发现和调用的服务器端工具，使应用不必为每个外部系统设计一套专用集成方式[9][10]。在服务器性能测试场景中，MCP 可以被理解为智能编排层与执行系统之间的能力边界。Perfa 的 MCP Server 并不直接执行压测，而是将上层请求转换为对 Node Agent 的 HTTP 调用，并负责服务器信息、Agent 生命周期、工具管理、Benchmark 任务和报告生成等能力的封装。该设计与 MCP 的协议思想一致，也符合复杂系统中分层解耦的工程原则。

从国内本科毕业设计和工程项目实现角度看，服务器性能测试平台常见方案仍以脚本封装、Web 管理页面或监控大屏为主。这类系统可以解决部分工具调用和结果展示问题，但在自然语言交互、智能任务编排、测试证据关联和报告生成方面仍有提升空间。由于中文文献需要以知网等数据库中的真实条目为准，本文后续会在参考文献整理阶段补充国内关于服务器性能测试、自动化运维和智能运维系统的研究资料。当前绪论中的国内现状分析不写入未经核验的具体文献，避免出现不可追溯引用。

综上，现有研究和工程实践为本文提供了三条技术基础：性能测试工具提供底层测量能力，可观测性技术提供运行证据采集能力，大语言模型与工具调用协议提供自然语言驱动的任务编排能力。然而，将三者整合为面向服务器性能测试的完整平台仍需要解决多个工程问题，包括测试工具生命周期管理、节点执行端设计、上层 Agent 与工具层的协议边界、流式交互反馈、报告证据保存和前端可视化呈现。Perfa 正是在这一背景下进行设计和实现的。

## 1.3 研究内容与主要工作

本文以 Perfa 项目为研究对象，围绕自然语言驱动的服务器性能测试平台开展设计与实现。系统目标不是提出新的性能测试算法，而是构建一套可运行的软件平台，将测试请求理解、工具能力封装、节点执行、监控采集、结果分析和报告展示整合到统一流程中。为了实现该目标，本文在需求分析、系统架构、模块实现和测试验证等方面展开工作。

在需求分析方面，本文从服务器性能测试的实际流程出发，分析用户在工具安装、参数配置、测试执行、结果查询和报告整理中的操作负担。系统需要支持用户通过自然语言描述测试目标，也需要允许前端页面展示服务器、报告和监控信息。由于性能测试涉及真实节点操作，系统还必须将自然语言编排与底层执行隔离，通过明确的工具接口完成受控调用。基于这些约束，本文将系统需求划分为服务器管理、Agent 管理、工具管理、Benchmark 执行、监控采集、报告生成和前端交互等功能。

在总体架构方面，本文设计了 WebUI V2、LangChain Agent、MCP Server 和 Node Agent 四层结构。WebUI V2 作为交互层，负责对话输入、工作流进度、服务器列表、测试报告和监控页面展示。LangChain Agent 作为智能编排层，对外提供 OpenAI 兼容接口，并根据用户请求调用工作流引擎和 MCP 工具适配器。MCP Server 作为能力封装层，将服务器注册、Agent 部署、工具安装、Benchmark 调度和报告生成封装为标准 Tool。Node Agent 作为节点执行层，直接运行在被测服务器上，负责监控指标采集、工具生命周期管理和压测任务执行。四层之间通过 HTTP、SSE 和 MCP 适配机制连接，形成从用户请求到真实执行的闭环。

在详细实现方面，本文重点完成 Node Agent、MCP Server、LangChain Agent 和 WebUI V2 四个子系统。Node Agent 使用 Flask 暴露健康检查、系统状态、工具管理、Benchmark 和存储日志等接口，并通过 Prometheus 指标端口提供 CPU、内存、磁盘和网络等监控数据。MCP Server 使用 MCP SDK 和 Starlette 对外提供 SSE 接入，将多个业务能力注册为 Tool，并通过 AgentClient 调用 Node Agent。LangChain Agent 使用 FastAPI 提供 `/v1/chat/completions` 等 OpenAI 兼容接口，支持同步和流式返回，并通过工作流场景处理快速测试、CPU 专项、存储专项、网络专项和综合评估等请求。WebUI V2 基于 Next.js、TypeScript、Ant Design、Tailwind CSS 和 ECharts 构建，提供对话、服务器、报告和监控页面。

在报告生成方面，本文将其作为体现系统闭环能力的重要模块。Perfa 的报告链路既包括 MCP Server 中的结构化报告工具，也包括 LangChain Agent 工作流中的 LLM 辅助总结和报告持久化。报告数据不仅保存最终 AI 结论，还保存原始测试结果、任务 ID、错误记录、工具调用和 Trace ID，使前端报告页面能够同时展示总结性内容和可回溯证据。对于尚未完全实现的历史测试向量检索和 RAG 增强报告，本文将其作为扩展设计进行讨论，并明确其当前属于预留能力，不将其描述为已完成的运行时功能。

在测试验证方面，本文将围绕功能正确性和端到端流程可用性展开。测试内容包括 Node Agent 健康检查、监控指标采集、测试工具安装与校验、Benchmark 任务提交和结果查询、MCP Tool 调用、LangChain Agent 对话接口、WebUI 页面联调和报告展示。通过这些测试，可以验证 Perfa 是否能够将自然语言请求转换为真实测试任务，并将执行结果回传到上层对话和报告页面。由于性能测试结果与硬件环境强相关，本文在结果分析中将重点说明测试环境、测试参数和可复现条件，避免脱离环境对数值作绝对化评价。

## 1.4 论文结构安排

本文围绕 Perfa 的需求分析、系统设计、模块实现和测试验证组织内容。第一章为绪论，介绍服务器性能测试平台的研究背景、相关技术发展现状、本文研究内容和论文结构。第二章介绍相关技术与需求分析，说明服务器性能测试、可观测性、大语言模型智能体、MCP、后端服务和前端技术栈的基本原理，并给出系统功能需求与非功能需求。第三章进行系统总体设计，描述 Perfa 的分层架构、模块职责、核心数据流、部署拓扑和接口关系。第四章展开系统详细设计与实现，围绕 Node Agent、MCP Server、LangChain Agent、WebUI V2、报告生成和知识增强设计说明关键实现。第五章进行系统测试与结果分析，设计测试环境和测试用例，对系统功能、接口、工作流和报告展示进行验证。第六章总结全文工作，分析系统不足，并提出后续优化方向。
