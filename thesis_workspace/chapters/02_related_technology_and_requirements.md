# 第二章 相关技术与需求分析

## 2.1 服务器性能测试技术基础

服务器性能测试的目标是通过可控负载观察服务器在计算、存储、内存和网络等资源维度上的表现。不同资源类型具有不同的瓶颈形态和评价指标，因此难以用单一工具覆盖所有测试需求。CPU 测试通常关注计算吞吐、单核能力、多核并行能力和特定算法执行效率；内存测试关注带宽、访问延迟和 NUMA 相关影响；磁盘测试关注 IOPS、吞吐量和读写延迟；网络测试关注吞吐、往返时延、丢包和重传。若测试平台只提供简单命令执行功能，用户仍需要自行理解工具差异和结果字段，平台对降低操作门槛的帮助有限。

Perfa 在节点执行层对测试工具进行了统一管理。源码中的 `ToolManager` 注册了 UnixBench、SuperPi、sysbench、OpenSSL、stress-ng、7z、STREAM、MLC、fio、hping3 和 iperf3 等工具，覆盖 CPU、内存、磁盘和网络多个类别。该设计并不是将所有工具混合为一个黑盒接口，而是保留工具自身的测试边界，同时通过统一的安装、卸载、状态检查和功能校验方法降低上层调用复杂度。对于上层系统而言，工具是否存在、二进制路径在哪里、版本是否可用、安装是否成功，都可以通过统一接口获取。对于节点执行端而言，不同工具仍可以由独立类负责安装逻辑和校验逻辑，便于后续加入新的测试程序。

测试任务执行比工具管理更复杂。性能测试并非普通的短请求，部分任务持续时间较长，执行过程中需要记录状态、保存日志、处理超时和收集结果。Perfa 的 `BenchmarkExecutor` 将任务生命周期划分为准备、运行、完成、失败等状态，并通过 Runner 机制适配不同测试工具。执行器在任务启动前检查工具是否安装，创建工作目录和日志文件，在执行时启动子进程并持续读取输出，在任务结束后调用结果解析逻辑。源码中还使用可重入锁管理当前任务，保证同一节点同时只运行一个 Benchmark 任务，避免多个压测任务互相干扰。这一约束符合性能测试的基本要求，因为并发压测会争用 CPU、磁盘和网络资源，使结果难以解释。

Runner 机制是 Perfa 测试执行层的关键抽象。每个 Runner 负责构建具体测试命令、准备运行环境、设置超时时间和解析输出结果。这样可以避免在统一执行器中写入大量工具分支逻辑，也使单个测试工具的维护范围更加清晰。以 fio、STREAM、UnixBench 和 iperf3 等工具为例，它们的参数形式、输出格式和运行时长差异较大，如果没有 Runner 抽象，上层系统很容易被工具细节污染。通过将工具执行封装为统一任务模型，Perfa 可以把“执行某类测试”抽象为 test name、params、task id、status 和 result 的组合，为 MCP Tool 和前端报告模块提供稳定的数据基础。

服务器性能测试还需要考虑结果解释边界。相同工具在不同硬件、内核参数、磁盘类型、网络拓扑和负载背景下得到的数值并不具备直接可比性。测试平台可以帮助用户统一流程、保存数据、生成报告，但不能脱离环境对结果作绝对判断。因此，Perfa 的测试设计更适合围绕同一服务器的多次对比、同类服务器之间的相对比较，以及测试过程中指标变化的解释。论文后续测试章节也应遵循这一原则，说明测试环境、测试参数和执行条件，再分析结果。

## 2.2 监控与可观测性技术

性能测试结果只有与运行时状态结合，才具备较强解释能力。单次 Benchmark 输出能够说明测试程序在某一条件下得到的指标，但无法完整解释测试期间系统资源是否饱和、是否出现异常负载、是否存在内存压力或磁盘 I/O 波动。Prometheus 将监控指标组织为时间序列，适合对 CPU、内存、磁盘和网络等指标进行周期采集与查询[3]。OpenTelemetry 则从 traces、metrics 和 logs 等信号出发，为系统观测数据的生成、采集和导出提供统一思路[4]。这些技术共同说明，测试平台需要将任务执行与运行证据关联起来，而不是只保存测试工具输出。

Perfa 的 Node Agent 在启动时同时初始化监控模块、Prometheus 指标服务和 HTTP API 服务。源码中 `NodeAgent.start()` 依次初始化工具管理器、BenchmarkExecutor、Prometheus metrics 服务、Monitor 后台线程和 Flask API。监控模块默认每 5 秒采集一次 CPU、memory、disk 和 network 指标，Prometheus metrics 服务默认暴露在 8000 端口。该设计使节点执行端不仅能够运行测试任务，还能持续暴露被测节点的资源状态。对于上层系统而言，测试任务和监控指标来自同一节点 Agent，减少了跨系统对齐数据的难度。

Perfa 的运行链路中还包括 VictoriaMetrics 和 Grafana。VictoriaMetrics 用于接收和保存 Prometheus 兼容指标，Grafana 用于可视化展示。项目运行文档中将监控链路描述为 `node_agent:8000 -> victoriametrics:8428 -> grafana:3000`，这说明系统将节点指标采集、时序存储和可视化展示进行了分层处理。论文中不应把这部分写成复杂的分布式追踪平台，但可以明确指出：Perfa 已经具备性能测试过程监控的基础链路，能够为测试结果解释和报告生成提供运行时上下文。

日志也是可观测性的重要组成部分。Node Agent、MCP Server 和 LangChain Agent 均在代码中配置了日志输出路径，并支持环境变量覆盖。BenchmarkExecutor 在执行任务时会创建日志文件，记录实际运行命令和输出内容。MCP Server 在工具注册和工具调用失败时记录日志，LangChain Agent 在工作流执行、场景路由和报告持久化过程中记录状态。日志、指标和任务结果共同构成 Perfa 的证据体系。当前系统没有把所有证据信号统一为完整的追踪图谱，但已经为后续 Trace ID 关联、错误诊断和报告回溯提供了基础。

在论文写作中，可观测性部分需要把握两个层次。已实现层面，Perfa 具备节点资源指标采集、Prometheus 指标暴露、VictoriaMetrics 存储、Grafana 展示和日志记录能力。扩展层面，代码中存在 OpenTelemetry 接入逻辑，Node Agent 和 MCP Server 在检测到相关环境变量或依赖时可以初始化追踪能力。由于这部分能力依赖运行环境配置，论文中应表述为“系统预留并部分接入可观测性链路”，而不是宣称所有组件都已经完整实现端到端追踪。

## 2.3 MCP 工具协议与能力封装

在自然语言驱动的性能测试平台中，模型编排层不能直接执行系统命令。若将 SSH 登录、工具安装、压测执行和报告生成全部写入大语言模型提示词或后端临时代码，系统会难以维护，也不利于权限控制和错误追踪。MCP 的核心价值在于提供一种标准化的工具能力暴露方式，使上层 AI 应用能够通过统一协议访问外部系统[9][10]。在 Perfa 中，MCP Server 位于智能编排层与节点执行层之间，负责将平台能力组织为可发现、可调用、可扩展的 Tool。

Perfa 的 MCP Server 基于 MCP SDK 实现，对外暴露 `/sse` 和 `/messages/` 两类入口。`/sse` 负责建立 MCP SSE 连接，`/messages/` 负责处理客户端提交的工具调用消息。服务器启动时会初始化本地数据库对象，并注册多组工具，包括服务器管理、Agent 生命周期管理、测试工具管理、Benchmark 压测管理和智能报告生成。工具注册逻辑集中在 `MCPServer._register_tools()` 中，上层调用时可以先获取工具列表，再根据工具名称和参数 schema 发起调用。与直接暴露大量 HTTP 接口相比，MCP Tool 的形式更适合被 Agent 编排层加载和调用。

MCP Server 的工具分组对应了 Perfa 的主要业务能力。服务器管理工具负责注册服务器、查询服务器列表、删除服务器和更新服务器信息；Agent 生命周期工具负责部署节点 Agent、检查状态、获取日志、配置和卸载；工具管理类 Tool 负责安装、卸载、列出和校验压测工具；Benchmark 类 Tool 负责运行压测、查询状态、取消任务、获取结果和列出历史；报告类 Tool 负责生成单次、对比或诊断报告。这种分组方式使系统能力边界清晰，上层 Agent 不需要理解 Node Agent 内部路由，只需要根据场景选择合适 Tool。

Perfa 的 MCP Server 不直接执行压测，而是通过 `AgentClient` 调用 Node Agent 的 HTTP API。该设计避免了能力封装层承担过多执行职责。MCP Server 管理服务器记录和工具调用协议，Node Agent 负责真实系统操作。若后续需要支持远程节点、混合部署或多节点比较，只需扩展服务器管理和 AgentClient 调用逻辑，而不必重写上层 Agent 工作流。论文第三章和第四章将继续围绕这一分层边界展开设计说明。

LangChain Agent 侧通过 `MCPToolAdapter` 连接 MCP Server。适配器会建立临时 SSE 连接，加载 MCP 工具列表，并将工具 schema 包装为 LangChain 兼容工具。工具调用时，适配器过滤空参数，建立连接，调用对应 MCP Tool，并解析返回结果。该模块解决了“标准 MCP Tool”和“LangChain Agent 工具接口”之间的格式转换问题。没有这一适配层，Agent 编排逻辑需要直接处理 MCP SDK 的连接、初始化、工具列表和结果解析，系统耦合度会显著提高。

## 2.4 大语言模型 Agent 与工作流编排

大语言模型在 Perfa 中承担的是交互理解和流程编排职责，而不是底层测试执行职责。LangChain Agent 后端基于 FastAPI 实现，对外提供 OpenAI 兼容接口，核心端点为 `/v1/chat/completions`。该接口既支持同步响应，也支持流式响应。流式模式下，最终回答和过程事件可以分开传输，前端能够展示工具调用、工作流进度和最终结论。这种设计比普通问答接口更适合长流程任务，因为性能测试往往需要经历环境检查、任务执行、结果收集和报告生成等多个阶段。

Perfa 的工作流路由模块将用户请求划分为多个场景。源码中的 `ScenarioRouter` 定义了 quick_test、full_assessment、cpu_focus、storage_focus、network_focus 和 free_chat 等场景，并使用关键词快速匹配与 LLM 路由结合的方式选择执行路径。关键词匹配适合处理明显请求，例如“测试磁盘性能”“跑个 fio”“全面评估服务器性能”；LLM 路由用于处理表达更自然或更模糊的请求。路由结果包含场景名称、置信度和选择原因，置信度低于阈值时回退到自由对话，从而降低误触发真实测试任务的风险。

工作流编排的意义在于将复杂测试任务拆解为可管理节点。以存储专项测试为例，系统需要识别目标服务器、检查 Agent 状态、确认 fio 或相关工具是否可用、提交 Benchmark 任务、轮询任务状态、收集结果并生成报告。若每次都由模型自由决定下一步，系统行为会难以预测。通过预定义场景图和节点函数，Perfa 可以把常见测试流程固化为可追踪路径，同时在报告生成和异常处理阶段利用 LLM 的语言总结能力。这样既保留了自然语言交互的灵活性，又避免模型直接控制底层命令。

近年来围绕 coding agent 的工程实践提出了 harness engineering 的概念，即通过指令、状态、验证、范围和生命周期约束，让长时间运行的 Agent 在可控边界内完成任务[21][22][23]。OpenAI 开源的 Symphony 项目展示了另一类典型形态：服务从 issue 系统读取工作项，创建隔离工作区并编排 coding agent 完成代码任务[24]。Perfa 借鉴的是这种工程方法，而不是照搬其产品形态。对于服务器性能测试而言，Agent 的工作对象不是 Git 仓库和 pull request，而是被测服务器、压测工具、监控指标和报告证据。因此，Perfa 更适合被理解为面向性能测试场景的垂直 Agent Harness：它通过固定生命周期约束每次测试都经历目标服务器选择、Agent 检查、工具检查、Benchmark 执行、结果获取、报告生成和证据保存，避免模型在长流程中自由发挥或提前宣布完成。

报告生成体现了 Agent 编排层的综合作用。Perfa 的工作流节点会优先尝试调用 MCP Server 中的 `generate_report` 工具，若工具不可用或缺少有效 server_id，则降级为 LLM 根据已有结果生成 Markdown 报告。工作流完成后，`AgentOrchestrator` 会把报告持久化到本地 JSON 存储中，报告对象包括场景、服务器、创建时间、状态、摘要、AI 报告、原始结果、错误记录、任务 ID、工具调用、Trace ID 和知识库检索片段。该设计使报告不只是最终文本，而是测试流程的结构化归档。后续若加入历史报告检索、相似测试召回或 RAG 增强生成，也可以基于这些结构化字段扩展。

大语言模型引入系统后也带来可靠性边界。模型生成的解释可能受到输入上下文限制，不能被当作唯一事实来源。Perfa 因此将真实结果保留在 raw results、task ids、tool calls、logs、trace id 和 knowledge matches 中，使报告页面能够同时展示 AI 结论、知识来源和原始证据。论文后续章节会将这一点作为系统设计原则：模型负责组织、解释和总结，工具与节点执行层负责事实数据，前端负责同时展示结论和证据。

## 2.5 Web 前端与交互技术

Perfa 的 WebUI V2 是用户主要入口。项目使用 Next.js 14 App Router、React、TypeScript、Ant Design、Tailwind CSS、Zustand 和 ECharts 等技术构建。Next.js 提供页面路由和 API 代理能力，TypeScript 提供类型约束，Ant Design 提供表单、卡片、抽屉、标签页和描述列表等常用组件，Tailwind CSS 用于样式组织，Zustand 用于前端状态管理，ECharts 用于图表展示。对毕业设计而言，这一技术栈能够支撑较完整的 Web 应用实现，而不是简单静态页面。

前端主页面包含对话、服务器、报告和监控四个导航入口。对话页负责用户自然语言输入、聊天消息展示、工作流进度展示和结构化结果卡片；服务器页负责服务器信息展示和 Agent 操作入口；报告页负责展示工作流综合报告和历史测试结果；监控页负责展示节点运行指标。这样的页面划分与后端能力保持一致：对话页对应 LangChain Agent 编排，服务器页对应 MCP Server 的服务器管理和 Agent 生命周期工具，报告页对应报告存储和报告详情接口，监控页对应 Node Agent 指标链路。

WebUI V2 并不直接访问 Node Agent 或 MCP Server，而是通过 Next.js API 代理访问 LangChain Agent 后端。前端 `api.ts` 中定义了聊天请求、服务器信息、报告列表、报告详情、Trace 摘要和会话管理等类型与调用函数。通过代理层访问后端可以减少浏览器端跨域和密钥暴露问题，也便于未来统一鉴权和请求日志。对于流式对话，前端还需要解析 SSE 数据，将最终回答与 workflow progress 等 metadata 区分展示。该设计使用户能够在长时间测试任务执行期间观察系统进度，而不是等待一个无反馈的最终响应。

报告页面是 Perfa 前端体现测试闭环的重要部分。页面会展示报告数量、报告类型、场景标签、执行状态、服务器信息和摘要，并在详情抽屉中展示 AI 结论、原始结果、任务 ID、错误记录和原始工具调用。这个设计与后端报告对象字段一致，使用户既能阅读自然语言总结，也能查看底层证据。对于指导教师或答辩场景而言，报告页面能够直观说明系统不是单纯聊天界面，而是包含真实测试结果、工作流归档和证据回溯的工程平台。

## 2.6 系统功能需求分析

结合服务器性能测试流程和 Perfa 已实现模块，系统功能需求可以归纳为七类。每类需求都对应明确的用户操作或后端能力，不以抽象概念堆砌为主。

| 需求编号 | 功能需求 | 主要说明 | 对应模块 |
|---|---|---|---|
| FR-1 | 服务器管理 | 支持注册、查询、更新和移除被测服务器，保存服务器 IP、别名、Agent 状态等信息。 | MCP Server、WebUI V2 |
| FR-2 | Node Agent 生命周期管理 | 支持部署 Agent、检查 Agent 状态、获取日志、配置和卸载 Agent。 | MCP Server、Node Agent |
| FR-3 | 测试工具管理 | 支持 CPU、内存、磁盘、网络等测试工具的安装、卸载、状态检查和功能校验。 | Node Agent、MCP Server |
| FR-4 | Benchmark 任务执行 | 支持提交压测任务、查询任务状态、取消任务、获取测试结果和查看历史任务。 | Node Agent、MCP Server |
| FR-5 | 自然语言测试入口 | 支持用户通过对话描述测试目标，由系统识别场景并调用对应工作流。 | LangChain Agent、WebUI V2 |
| FR-6 | 报告生成与归档 | 支持根据测试结果生成结构化报告，保存摘要、AI 结论、原始结果、错误记录和工具调用。 | LangChain Agent、MCP Server、WebUI V2 |
| FR-7 | 监控展示 | 支持采集和展示 CPU、内存、磁盘、网络等运行指标，为结果分析提供上下文。 | Node Agent、VictoriaMetrics、Grafana、WebUI V2 |
| FR-8 | Agent Harness 生命周期约束 | 每次自然语言性能测试必须经过服务器选择、Agent 检查、工具检查、Benchmark 执行、结果获取、报告生成和证据保存。 | LangChain Agent、MCP Server、Node Agent |

服务器管理是系统所有能力的入口。用户需要先将目标服务器加入系统，后续 Agent 部署、工具管理和压测任务才能定位到具体节点。服务器记录还需要保存 Agent 端口、状态和硬件信息，便于前端判断服务器是否可测试。该需求对应 MCP Server 的服务器管理工具和前端服务器页面。

Node Agent 生命周期管理解决的是远程执行端可用性问题。性能测试最终需要在被测节点上运行，平台必须知道 Agent 是否已部署、服务是否健康、日志是否可读取。Perfa 的 Agent 生命周期工具通过部署、状态检查、日志获取和卸载等能力，将节点端运维操作封装到 MCP Tool 中。该需求对系统可用性影响较大，因为 Agent 不可达时，上层对话和报告生成都无法获得真实测试数据。

测试工具管理用于降低环境准备成本。不同服务器可能缺少 fio、stream、iperf3 等工具，测试前需要检查工具状态并在必要时安装。Perfa 将工具生命周期放在 Node Agent 中处理，使每个节点可以根据自身环境管理工具。MCP Server 再将这些能力暴露给上层 Agent，实现“用户发起测试请求后，系统先检查工具是否可用”的流程。

Benchmark 任务执行是平台核心功能。系统需要支持同步短测和异步长测，能够返回 task id，允许后续查询状态和结果。由于性能测试具有资源独占性，Node Agent 需要限制同一时间运行的任务数量，避免并发任务互相干扰。任务执行完成后，系统还需要保存结果和日志，为报告生成和结果复查提供依据。

自然语言测试入口体现 Perfa 与传统测试平台的差异。用户不必直接选择工具和填写所有参数，而是通过对话描述目标，例如“测试这台服务器的磁盘性能”或“做一次全面评估”。系统通过场景路由和工作流将自然语言转化为具体步骤。该需求要求后端同时具备对话接口、场景识别、工具调用和过程事件返回能力，前端则需要展示流式进度和最终结果。

报告生成与归档需求用于形成测试闭环。测试结果如果只停留在接口返回中，后续很难复盘。Perfa 将工作流结果保存为报告对象，并在报告页展示。报告内容需要包含自然语言结论，也需要保留原始结果、错误记录、任务 ID、工具调用、Trace ID 和知识库片段等证据。对于后续扩展，系统还可以在报告库基础上加入历史测试对比、相似结果检索和知识增强生成。

Agent Harness 生命周期约束是 Perfa 从通用 AI 工程实践中吸收的需求。长流程 Agent 容易在中间状态不完整时给出总结，性能测试场景不能接受这种行为，因为缺少 task_id、原始结果或日志路径的结论无法复核。系统因此需要把“选择服务器、检查 Agent、检查工具、执行 Benchmark、获取结果、生成报告、保存证据”固化为工作流步骤，并将缺失证据作为报告风险呈现。该需求使 Perfa 与单纯聊天机器人区分开来，也使论文中的“智能化”建立在真实执行链路之上。

监控展示需求用于补充 Benchmark 结果解释。测试过程中 CPU 使用率、内存占用、磁盘 I/O 和网络流量等指标可以帮助判断瓶颈来源。Perfa 通过 Node Agent 暴露 Prometheus 指标，并通过 VictoriaMetrics 和 Grafana 提供指标存储与可视化基础。WebUI V2 的监控页面则承担集成展示入口，使用户在同一平台中查看测试与监控信息。

## 2.7 系统非功能需求分析

除功能需求外，Perfa 还需要满足可扩展性、可维护性、可靠性、安全边界、可观测性和易用性等非功能需求。这些需求决定系统能否从毕业设计原型发展为可持续维护的平台。

| 需求编号 | 非功能需求 | 设计约束 | Perfa 中的体现 |
|---|---|---|---|
| NFR-1 | 可扩展性 | 新增测试工具或工作流时不应大规模修改既有代码。 | Runner 机制、ToolManager 注册、MCP Tool 分组、Workflow 场景 |
| NFR-2 | 可维护性 | 各层职责清晰，故障定位时能找到对应模块。 | WebUI、LangChain Agent、MCP Server、Node Agent 四层结构 |
| NFR-3 | 可靠性 | 长任务需要状态管理、超时处理、日志记录和错误返回。 | BenchmarkExecutor 任务状态、日志文件、异常处理 |
| NFR-4 | 安全边界 | 大语言模型不能直接执行系统命令，外部调用需要受控接口。 | MCP Tool 封装、Node Agent API、MCP API Key |
| NFR-5 | 可观测性 | 系统需要记录指标、日志、工具调用和报告证据。 | Prometheus metrics、日志、Trace ID、报告 raw results |
| NFR-6 | 易用性 | 用户应能通过自然语言和 Web 页面完成主要操作。 | 对话页、服务器页、报告页、监控页 |
| NFR-7 | 证据完整性 | 系统不得只保存 AI 结论，关键测试证据缺失时需要可见。 | task_ids、raw_results、tool_calls、trace_id、knowledge_matches |

可扩展性主要体现在工具和场景两个方向。测试工具扩展依赖 ToolManager 和 Runner 机制，新工具可以通过新增工具类和 Runner 类接入。场景扩展依赖工作流路由和场景图，新的测试场景可以在已有 quick_test、cpu_focus、storage_focus 等基础上增加。MCP Tool 分组则为上层 Agent 提供稳定能力边界，使新能力可以以工具形式暴露。

可维护性来自分层结构。WebUI V2 负责展示和交互，LangChain Agent 负责意图识别与工作流，MCP Server 负责工具协议和业务能力封装，Node Agent 负责真实节点执行。不同层之间通过 HTTP、SSE 和 MCP 连接，既保留了通信边界，也方便在故障发生时定位问题。例如前端无法显示报告时，可以沿着 WebUI API、LangChain reports 接口、ReportStore 和工作流持久化逻辑逐层排查。

可靠性需求在压测系统中尤其重要。Benchmark 任务可能耗时较长，也可能因为工具缺失、参数错误、权限不足或超时失败。Perfa 的执行器在任务开始前检查工具状态，执行时保存日志和输出，超时时终止进程，失败时返回错误信息。虽然当前系统仍有进一步强化任务队列和并发调度的空间，但已有实现能够支撑本科毕业设计所需的完整任务生命周期。

安全边界需要谨慎描述。Perfa 当前通过 MCP API Key、受控 Tool 调用和 Node Agent API 将模型与系统命令隔离，避免大语言模型直接执行任意 shell 命令。该设计提供了基本边界，但还不能称为完整安全体系。若面向生产环境，还需要补充更严格的用户鉴权、权限分级、命令白名单、审计日志和敏感信息保护。论文中应将当前安全能力定位为“接口级访问控制和执行边界设计”，而不是完整安全防护方案。

可观测性需求贯穿系统各层。Node Agent 暴露 metrics，BenchmarkExecutor 保存任务日志，MCP Server 记录工具注册和调用异常，LangChain Agent 保存工作流报告，前端展示报告、Trace ID 和原始工具调用。这些信息可以帮助用户解释测试结果，也能帮助开发者排查系统问题。后续若接入完整 OpenTelemetry 链路，可以进一步把一次用户请求、工具调用和节点执行关联为端到端追踪。

易用性是自然语言测试平台的核心目标。用户不应被迫记住所有工具命令和后端接口，而应通过对话或页面完成操作。Perfa 的前端四页面提供了较清晰的入口，对话页处理自然语言请求，服务器页管理被测节点，报告页查看测试归档，监控页观察运行状态。该设计降低了系统学习成本，也便于答辩演示时展示完整流程。

## 2.8 本章小结

本章围绕 Perfa 的技术基础和系统需求进行了分析。服务器性能测试技术决定了系统必须覆盖多类测试工具，并对工具生命周期和任务执行过程进行统一管理。监控与可观测性技术说明测试结果需要与运行指标、日志和错误证据结合，才能形成可解释结论。MCP 工具协议为智能编排层和节点执行层之间提供了标准化能力边界，大语言模型 Agent 与工作流机制则使自然语言请求能够被转换为可执行测试流程。WebUI V2 承担用户交互、报告展示和监控入口，使系统从后端工具集合发展为完整平台。

在需求层面，Perfa 需要支持服务器管理、Agent 生命周期管理、测试工具管理、Benchmark 执行、自然语言测试入口、报告生成与归档、监控展示等功能，同时满足可扩展性、可维护性、可靠性、安全边界、可观测性和易用性等非功能要求。这些需求为第三章系统总体设计提供了依据，后续章节将围绕四层架构、核心数据流和模块实现展开说明。
