## 使用规则

- 每次用户在正文中使用 `@cmt` 或在消息中提出总体问题时，将评论原文和处理决策记录在本文件。
- 正文修改后必须删除对应章节中的 `@cmt` 内容。
- 同类问题后续章节继续沿用，除非用户明确改变要求。

## 2026-05-03 - 00 摘要

### 正文内评论

1. “我们的性能测试 only 关于一个不跑负载的服务器，所以和故障定位没关系吧”
   - 处理决策：摘要开头删除“故障定位”，改为“上线验收、硬件资源评估和容量规划”等更符合单服务器性能测试的表述。

2. “不是基于 Perfa，Perfa 就是我的毕设项目”
   - 处理决策：删除“基于 Perfa 项目”，改为“本文设计并实现了一套……平台”，将 Perfa 作为系统名称或项目名称，不写成外部基础项目。

3. “先说中文是啥，然后括号是英语吧，我们这个东西再摘要的时候没让知道这个是什么”
   - 处理决策：摘要中先使用中文逻辑层级描述，再在括号中给出英文或缩写；弱化 WebUI V2、LangChain Agent、Node Agent 等具体实现名称。

4. “项目才能实现吧，文字不能”
   - 处理决策：将“本文重点实现了”改为“项目重点实现了”或“系统实现了”，避免把论文文本当作实现主体。

### 总体评论

1. “可观测性”关键词颗粒度过大，正文如果只是指标采集，关键词不应使用可观测性。
   - 处理决策：摘要和关键词中将“可观测性”改为“监控采集”或“运行监控”，避免 Metrics、Logging、Tracing 全栈语义过大。

2. 摘要架构描述技术堆栈化，WebUI V2、LangChain Agent、Node Agent 等版本号或框架名降低论文抽象层级。
   - 处理决策：中文摘要采用“交互层、智能编排层、能力封装层、节点执行层”描述，括号中保留必要英文概念；英文摘要使用 interaction layer, intelligent orchestration layer, capability encapsulation layer, node execution layer。

3. “具备基础能力”评价太弱，像工程能跑，缺少可验证结论。
   - 处理决策：改为“验证了系统在服务器注册、工具管理、Benchmark 执行、报告归档和页面交互等流程中的可用性”，不虚构效率提升或准确率数据。

4. 前端契约测试不是性能测试平台论文的主要评价指标，摘要不宜突出。
   - 处理决策：摘要中改为“模块测试、接口检查和运行状态验证”，弱化“前端契约测试”。

5. “被测节点”“target node”“node execution layer”术语需统一。
   - 处理决策：中文统一使用“被测服务器/节点执行端”；英文统一使用 target server 和 node execution layer，避免 target node 与 node execution layer 混用。

6. “证据回溯”定义不明。
   - 处理决策：摘要中补充“保留原始 Benchmark 结果、任务标识和工具调用记录”等解释。

7. 摘要结尾写“稳定性、多节点管理仍需完善”容易显得完成度不足。
   - 处理决策：删除负面收尾，改为客观说明系统为后续扩展多节点调度和报告分析提供基础。
